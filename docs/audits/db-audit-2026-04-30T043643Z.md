## §1 — Executive summary

**Completeness (static-analysis only):** ~22 of 38 doc-committed schema items map cleanly to source = ~58 %. Live verification (extensions, row counts, partition state, trigger live-test, materialized-view freshness, prompt-versions active row) is entirely blocked.

**Severity counts:** 🔴 4 · 🟠 9 · 🟡 8 · 🔵 2

**Conflicts requiring resolution (§7):** 3

**Top three risks (plain English):**

1. **The audit hash chain — the entire tamper-evidence story the TDD sells in four separate places — is not in the schema.** No `audit_hash` column exists in `setup_database.sql` or in any migration. `service/services/audit.py` does not import `hashlib` or compute a hash. The trust substrate is documented but unimplemented.
2. **Partitioning migration silently breaks the messages PK contract and four downstream FKs.** Migration 001 changes `messages` PK from `(id)` to `(id, created_at)` to satisfy native partitioning, but `message_feedback`, `user_corrections`, `outcome_linkages`, and `ml_predictions` still FK-reference `messages(id)`. PostgreSQL requires the FK target to be unique on its own — after migration, `messages(id)` alone is not. Either the migration was run and those FKs were silently dropped, or the migration cannot complete on a populated database. Live DB needed to determine which.
3. **Hybrid-retrieval BM25 leg is doc-only.** TDD §5.1 commits to a generated `bm25_tsv TSVECTOR` column on `document_chunks` plus `idx_chunks_bm25_gin`. Neither exists. `setup_database.sql` only ships `idx_chunks_text_trgm` (pg_trgm trigram on raw `chunk_text`), which is a different mechanism. The hybrid retriever's keyword leg is running on trigram, not BM25/TSVECTOR, contrary to what the as-built doc claims.
