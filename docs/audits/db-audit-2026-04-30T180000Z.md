**Top three risks:**

1. **Migration 001's inline `CREATE TABLE messages` and `audit_log` DO-blocks omit the `audit_hash` column** that the canonical setup script and the audit service both depend on. On any environment that gets initialized via `setup_database.sql` and *then* runs migration 001 against an already-partitioned DB, the early-exit `IF is_partitioned THEN RETURN` guard saves us. But on any environment migrated *from* a pre-v3 baseline (legacy table → rename → recreate via migration 001), the recreated table will silently lack `audit_hash`, breaking the SHA-256 chain on first append. (F-26)
2. **`v_chat_perf_daily` is now a `MATERIALIZED VIEW` (good — fixes prior F-09) but no scheduled refresh is wired anywhere.** [service/main.py](service/main.py) `_nightly_jobs_loop` refreshes `v_rca_precision_daily` only; `v_chat_perf_daily` will trend stale forever in production unless an external cron or `pg_cron` schedule lands. (F-25)
3. **`failure_modes` seed still ships only 9 of the ~25 codes TDD §5.3 commits to.** Carry-forward of prior F-07; the seed file was untouched in the remediation pass. Any defect classified into a code not in `('delam_hotpull','delam_cold','off_tenter_edge_fold','bubble_adhesive','bubble_precoat','streak_frontback','cw_out_of_spec','contamination_other','other')` will be rejected by the FK. (F-07-CARRY)
